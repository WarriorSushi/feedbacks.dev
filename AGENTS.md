## What to optimize for

- extremely clear first-run setup
- developer trust in copy-paste installation
- fast time-to-value
- small, maintainable product surface
- strong information hierarchy over flashy UI

## Product thesis

`feedbacks.dev` should help a developer:
- install feedback collection in minutes
- collect useful context automatically
- triage feedback quickly
- route important issues into existing workflows

## Guardrails

- Keep the install path simple enough for a first-time user.
- Separate quick install from advanced customization.
- Avoid reintroducing bloated, all-in-one setup screens.
- If old archived behavior conflicts with current docs, follow the current docs.
- Treat docs in `docs/` as authoritative unless explicitly updated.

## Archived project usage

Use `archived_project/` only when it exists and is needed for:
- prior UX lessons
- implementation references
- asset extraction
- migration planning

Do not assume the archived design, architecture, or naming should be preserved.
