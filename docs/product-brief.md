# Product Brief

## Product name

`feedbacks.dev`

## One-line description

A developer-first feedback widget and triage dashboard that is fast to install, easy to trust, and designed to turn raw feedback into action.

## Problem

Teams know they should collect feedback inside their product, but most solutions are frustrating for at least one reason:

- installation is slower than expected
- widgets feel bulky or visually intrusive
- dashboards become cluttered inboxes with no workflow
- tools are expensive before a product has traction
- integrations are too shallow or too enterprise-heavy

For small teams, the result is usually one of these:
- no feedback tool at all
- a generic form linked from the footer
- scattered feedback across email, chat, screenshots, and support tickets

## Opportunity

There is room for a product that feels like:
- the install simplicity of analytics
- the signal quality of an in-product feedback tool
- the actionability of a lightweight issue triage layer

## Core promise

A developer can go from zero to collecting useful in-product feedback in under 10 minutes, without making their site feel slower, uglier, or harder to maintain.

## Target users

### Primary

- solo founders
- indie makers
- SaaS engineers
- developer-led startups

### Secondary

- product managers working with small engineering teams
- agencies maintaining websites or web apps for clients
- internal tooling teams that need lightweight feedback collection

## Why people would use it

- it is faster to install than a heavyweight product feedback suite
- it looks more intentional than a generic support widget
- it captures enough context to make feedback actionable
- it gives teams a clean triage workflow without requiring Jira-scale process
- it can fit early-stage products and still scale into a more mature workflow

## What makes it different

### 1. Setup simplicity

The install path must feel obvious:
- create project
- copy code
- see widget live
- customize later

### 2. Developer trust

The product has to feel dependable:
- copy-paste snippets work
- docs are consistent
- API naming is stable
- widget performance stays tight

### 3. Signal over noise

Feedback should come with context:
- page URL
- environment or product area
- optional screenshot
- optional category or sentiment

### 4. Actionability

The dashboard is not just storage. It should help users:
- identify recurring issues
- prioritize high-signal requests
- route important items into team workflows

## Product principles

- Fast first value beats feature density.
- Clarity beats visual novelty.
- Good defaults beat mandatory configuration.
- Structured feedback beats raw message dumps.
- Integrations should reduce workflow switching, not create more setup burden.

## Success criteria

The rebuild is successful if a new user can:

1. understand the product in under 30 seconds
2. install the widget in under 10 minutes
3. receive the first piece of feedback in under 30 minutes
4. triage that feedback without confusion

## Strategic note

The product still has to win on trust and usability, but the current release target is no longer a narrow MVP.
The launch target is a full-surface paid release with:
- Free and Pro tiers
- Dodo Payments for billing
- public boards, integrations, and AI/MCP treated as real product surfaces rather than future ideas
